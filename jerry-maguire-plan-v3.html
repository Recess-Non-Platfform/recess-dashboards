Architecture Decision Log

Five locked-in decisions that shape everything else.

1 Option C (Hybrid) Over Option B
Locked

The original PRD labels Option B (Replace Clay) as "recommended" but contradicts itself by using Clay in Phase 2. We're officially locking Option C (Hybrid): Clay stays for what it's uniquely good at, custom AI layer handles everything else.

Clay owns

  • Contact sync (LinkedIn Sales Nav → HubSpot)
  • 75+ provider enrichment waterfall
  • LinkedIn signals (job changes, posts)
  • RSS monitoring
  • NeverBounce verification
  • Periodic departure re-scrape

Jerry Maguire owns

  • List building (natural language → query)
  • Personalized draft generation (GPT-4o + BQ RAG)
  • BigQuery-powered case studies
  • Custom guardrails (fabrication prevention)
  • Voice matching per rep
  • Multi-channel orchestration + state management
  • Attribution tracking

Rationale: Clay's LinkedIn signal monitoring and 75+ provider waterfall cannot be cost-effectively rebuilt. Clay drops to enrichment-only tier (~$149–349/mo) instead of full orchestration tier (~$349–800/mo) — real savings, not fake savings from scope cuts.

2 HubSpot Sequences Dropped Entirely
Locked

Original PRD used HubSpot Sequences for the warm path. We're eliminating it because:

  • Cannot modify sequence steps via API (only enroll/unenroll)
  • One sequence per contact limit
  • Requires per-rep Sales Pro seat
  • Enrollment resets on unenroll/re-enroll
  • Splits sending infrastructure across two tools unnecessarily

Replacement: SmartLead with multiple sender types on a single account — burner domains for cold, rep real emails connected as senders for warm. Single sending infrastructure, same orchestration, same pause-on-reply, same API. Rep never touches SmartLead UI.

Gmail API scope narrowed to thread replies only (re-engagement + post-meeting recap). Net-new warm outreach goes through SmartLead with rep email senders.

3 Unified Dual-Mode Engine (Not Two Systems)
Locked

Jerry Maguire is mode-aware from day one. Every service accepts {mode: "demand" | "supply"} as a first-class parameter and loads per-mode configuration from Firestore modes/{demand|supply} documents.

Why one engine with two modes beats two separate systems:

  • Every improvement benefits both sides simultaneously (voice matching, guardrails, orchestration, attribution)
  • Supply side never lags demand side because dev shop attention can't drift
  • Reps share the same mental model — reduces training, adoption friction
  • ~15–25% extra effort up front vs. 100% extra effort building two systems
  • Unified Slack briefing per rep — reps see their own work regardless of mode
4 Supply ABM Is a Feeder, Not a Frontend
Locked

The Supply ABM Category Playbook Framework (Leo's separate workstream) produces strategy artifacts — Notion playbooks, Asana projects, Streamlit dashboards, Claude skills. The Claude skills output structured briefs that POST to Jerry Maguire's /potential-sends API — just like the demand natural language list builder does.

Separation of strategy from execution:

Strategy Layer

Human-led, Claude thought-partner

  • Who to target
  • What to say
  • What's the category play
  • Supply: Notion playbooks, Asana projects
  • Demand: PRD + recess-messaging.md + triggers

Execution Layer (Jerry Maguire)

Automated, mode-aware

  • Generate drafts (GPT-4o + BQ)
  • Route through channels
  • Manage state (pause on reply)
  • Track replies + attribution
  • Enforce guardrails
5 Dev Shop Owns Content + Logic (Not Just Infrastructure)
Locked

Prompt engineering and content quality are the single biggest determinants of rep adoption. Making them a first-class dev shop deliverable (Package 7) with dedicated ownership is much safer than squeezing them between you + Claude's other work.

RFP implication Dev shops must demonstrate prompt engineering capability in their proposal. Ask them: "Show us a sales email draft your system generated, alongside the prompt that produced it. Walk us through how you'd iterate if reply rates dropped 20% week-over-week." Dev shops that can't answer this well are wrong for this project — even if they're great at Cloud Run infrastructure.
