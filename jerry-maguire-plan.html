Section 01

Executive Summary

Jerry Maguire transforms Recess Digital's outbound sales from manual hustle to an AI-orchestrated engine. Starting with deal re-engagement in Week 1 (stale deal detection + AI-drafted Gmail replies), the system expands to monitor campaign completions, job changes, news signals, and supply opportunities — then auto-generates personalized, voice-matched email drafts for sales reps to review and approve. Approved sends route through SmartLead (cold email via 3 warmed burner domains), HubSpot Sequences (warm/existing contacts), or Heyreach (LinkedIn for 3-4 reps) — with a multi-channel state machine coordinating cross-platform cadences. Every email is enriched with contact intelligence from Clay, validated against guardrails, and managed through a HubSpot Tasks-based review queue (Professional tier — no custom objects). Campaign data migrates from Airtable to BigQuery to power blinded case study auto-generation. Reps keep full control: the AI drafts, humans decide.

Goal & Scope

🎯 One-Sentence Goal

Build an AI-powered outbound sales system that detects opportunities, generates personalized multi-channel outreach, and orchestrates sending — with human approval at every step — reducing rep manual effort by 70%+ while increasing pipeline quality.

✅ In Scope

  • Deal re-engagement system — 8-day staleness rule, Gmail draft replies on existing threads, 5-tactic playbook
  • AI email copy generation (GPT-4o) for 6 email types: case study, sales enablement, evergreen, signal-based, re-engagement, conference
  • 6 trigger types: campaign complete, job change, RSS news, LinkedIn post, supply opportunity, quarterly recap
  • Multi-channel send orchestration: SmartLead (cold, 3 burner domains), HubSpot Sequences (warm), Heyreach (LinkedIn, 3-4 reps)
  • Cross-channel state machine with pause-on-reply coordination

  • Mobile app for reps
  • Multi-tenant / white-label architecture

  2. "Brief Me" response time < 10 seconds from Slack command to intelligence one-pager
  3. Rep time savings of 2+ hours/week per rep on outbound prep
  4. AI draft approval rate > 60% (drafts sent as-is or with minor edits)
  5. Reply rate ≥ parity with manual outbound (baseline measured in Phase 1)
  6. Zero brand-name leaks in sent case study emails
  7. Full attribution tracking: trigger → draft → send → reply → meeting → deal

HubSpot Tasks-Based Queue — "Potential Send" (Professional Tier Fallback)
How Tasks replace Custom Objects: Each potential send creates a HubSpot Task assigned to the rep. Task subject = draft subject line. Task body = draft preview + link to full draft in Slack/Firestore. Tasks are associated to the relevant contact(s). Reps use HubSpot task views filtered by type to see their queue.

Draft Versioning

All draft versions are stored in Firestore. HubSpot always shows the latest. When a rep edits in HubSpot, a webhook fires to update Firestore.

Draft v1 → AI generates initial draft
Draft v2 → Auto-revision if guardrail check fails
Draft v3 → Rep edits in HubSpot (manual)
Draft v4 → Rep makes final tweaks
→ Status: "approved" → version locked, sent to SmartLead/HubSpot

Cold vs. Warm Routing Logic

Python — route_contact() Decision Function 
def route_contact(contact: dict) -> str:
    """Determine send channel for a contact."""

    # Active customer or open deal → always HubSpot (warm)
    if contact.get('has_open_deal') or contact.get('is_customer'):
        return 'hubspot_sequence'

    # Prior engagement in last 90 days → HubSpot (warm)
    if contact.get('last_engagement_date') and \
       days_since(contact['last_engagement_date']) < 90:
        return 'hubspot_sequence'

    # No valid email → LinkedIn only
    if contact.get('email_status') != 'verified':
        return 'heyreach_linkedin'

    # Default: cold via SmartLead
    return 'smartlead_cold'

Multi-Channel State Machine

When a contact has both email and LinkedIn, the orchestrator runs a coordinated cadence:

stateDiagram-v2 [*] --> Queued: Draft approved Queued --> LinkedIn_Connect: Day 1 LinkedIn_Connect --> Waiting_Accept: Request sent Waiting_Accept --> Email_1: Day 3 Waiting_Accept --> Connected: Accept received Connected --> LinkedIn_Message: Day 5 Email_1 --> Email_2: Day 6 (no reply) LinkedIn_Message --> Replied: Reply on LinkedIn Email_1 --> Replied: Reply on email Email_2 --> Replied: Reply on email Email_2 --> LinkedIn_Message_2: Day 8 LinkedIn_Message_2 --> Email_3: Day 11 Email_3 --> Completed: Sequence done Replied --> All_Paused: Pause all channels All_Paused --> Rep_Followup: Rep takes over
Cross-Channel Pause Logic — Python 
async def handle_reply(contact_id: str, channel: str, reply_data: dict):
    """Reply on one channel → pause all others immediately."""

    state = await firestore.get(f"orchestration/{contact_id}")

    # 1. Pause SmartLead if active
    if state['channels']['smartlead'] and \
       state['channels']['smartlead']['status'] == 'active':
        await smartlead.pause_lead(
            campaign_id=state['channels']['smartlead']['campaign_id'],
            lead_id=state['channels']['smartlead']['lead_id']
        )

    # 2. Pause Heyreach if active
    if state['channels']['heyreach'] and \
       state['channels']['heyreach']['status'] in ('active', 'waiting_accept'):
        await heyreach.toggle_lead(action='pause')

    # 3. Update HubSpot
    await hubspot.update_contact(contact_id, {
        'outbound_status': 'Replied',
        'outbound_reply_channel': channel,
        'outbound_reply_date': datetime.utcnow().isoformat()
    })

    # 4. Notify rep
    await slack.send_message(
        channel=state['rep_slack_id'],
        text=f"🎉 Reply from {state['contact_name']} via {channel}!"
    )

Content & Copy Generation Engine

The AI Model

Primary: OpenAI GPT-4o (via API, not ChatGPT UI)

  • Best-in-class for sales copy — consistently produces natural, non-robotic email language
  • 128K context window — full campaign data + contact profile + prior emails + brand research in one call
  • Function calling — structured JSON output for template fields (subject, body, CTA)
  • Cost: ~$2.50/1M input tokens, ~$10/1M output → ~$0.02–0.08 per email draft
  • Latency: 2–5 seconds per draft, acceptable for async queue-based generation

Fallback: Anthropic Claude 3.5 Sonnet — used if GPT-4o is rate-limited or down.

Voice Matching — How It Works

  1. Data collection: Pull rep's last 20–30 sent emails from BigQuery (email history already synced from HubSpot) or fallback to HubSpot Engagements API
    SELECT * FROM email_history WHERE direction='SENT' AND owner_id={rep_id} ORDER BY timestamp DESC LIMIT 30
  2. Style extraction: GPT-4o one-time "voice analysis" prompt:
    "Analyze these 20 emails. Extract: greeting style, sentence length tendency, sign-off preference, humor level, common phrases. Return a 100-word style guide."
  3. Caching: Style guide stored in Firestore (keyed by rep_id), refreshed monthly or after 10+ new emails
  4. Injection: Style guide + 3–5 raw email samples included in every draft prompt as VOICE SAMPLES

Cost: ~$0.03 for initial analysis. $0 ongoing (cached).

Python — Draft Validation Function 
def validate_draft(draft: dict, contact: dict, campaign: dict) -> tuple[bool, list[str]]:
    """Post-generation validation. Returns (is_valid, issues)."""
    issues = []

    # No dollar amounts
    if re.search(r'\$[\d,.]+', draft['body']):
        issues.append("DOLLAR_AMOUNT_DETECTED")

    # No real brand names in case study emails
    if campaign.get('type') == 'case_study':
        for brand in BRAND_BLOCKLIST:
            if brand.lower() in draft['body'].lower():
                issues.append(f"BRAND_NAME_LEAKED: {brand}")

    # Length checks
    if len(draft['subject']) > 60:
        issues.append("SUBJECT_TOO_LONG")
    if len(draft['body'].split()) > 200:
        issues.append("BODY_TOO_LONG")

    # Contact eligibility
    if contact.get('email_status') != 'verified':
        issues.append("EMAIL_NOT_VERIFIED")

    return (len(issues) == 0, issues)

If validation fails, the system auto-retries (up to 2 re-prompts) with specific issues flagged. If still fails → queued with warning flag for manual review.

Top 3 Risks (Ranked by Both Reviewers)

#1 — Rep Adoption

The system works perfectly but reps don't use it. Inertia beats innovation unless UX is 10x better.

Likelihood: High  |  Impact: Critical

Mitigations:

  • Ship visible value in Week 1 ("Brief Me" + digests)
  • HubSpot-native for all workflows — zero new tools
  • "What goes away" framing: show the 2–4h manual workflow vs. 15-min AI flow
  • Leaderboard/gamification (reps are competitive)
  • Weekly adoption check-ins for first 8 weeks

#2 — Multi-Channel Orchestration Complexity

Cross-channel dedup, error handling, and state management across three systems that don't natively talk.

Likelihood: Medium  |  Impact: High

Mitigations:

  • Email-first MVP — don't add LinkedIn until email is solid
  • Staged rollout (Phase 4 Stage A → Stage B)
  • Comprehensive state machine with explicit error states
  • Webhook retry logic + dead letter queue for failed events

#3 — Airtable → BigQuery Migration Complexity

Campaign data is in Airtable, not BigQuery. Case study auto-generation (Phase 4) is blocked until migration completes. Schema differences, data quality issues, and ongoing sync requirements could delay Phase 4.

Likelihood: Medium  |  Impact: High

Mitigations:

  • Start Airtable audit in Phase 2 (Week 4) — don't wait until Phase 4
  • Define minimum viable BigQuery schema (which fields are required for case study generation?)
  • If migration is complex: manual case study creation as Phase 4 fallback while ETL builds
  • Consider ongoing Airtable → BigQuery sync if Airtable stays as data entry point
